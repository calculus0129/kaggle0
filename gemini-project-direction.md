이거 좀 바꿔야할것 같은게 ML과 LLM이 생각보다 다르고 LLM이 ML 문제에 많이 쓰이진 않더라고... 어떻게 바꿔야할까?

Gemini's Answer

맞아요, 정확한 지적입니다! Kaggle에서 타이타닉 같은 **전형적인 머신러닝(ML)** 문제에 LLM을 쓰는 건 마치 "못을 박는 데 최첨단 레이저 커터를 사용하는 것"과 비슷하거든요. 효율도 떨어지고 성능도 XGBoost 같은 라이브러리에 밀리는 경우가 많죠.

실제 실무에서도 **정형 데이터(Tabular Data)**는 전통적인 ML이, **비정형 데이터(Text, Image)**나 **추론/변환**은 LLM이 담당하는 식으로 역할 분담이 확실합니다.

이 차이점을 명확히 인지하고 있다는 점을 강조하면서, **"적재적소에 맞는 기술 선택(Right Tool for the Right Job)"** 능력을 기르는 방향으로 계획을 수정해 보았습니다.

## 🚀 Kaggle로 익히는 데이터 솔루션: ML과 LLM의 적재적소 활용법 (0)

Kaggle 대회를 통해 **전통적 ML(예측/분류)**과 **현대적 LLM(이해/생성)**의 차이를 몸소 체험하고, 문제 유형에 따른 최적의 모델 선택 기준을 수립하여 공유하는 것을 목표로 합니다.

**유형:** ① 신규 아이템 계획 - 실무 데이터 해결 역량 강화

### 🔍 Background (배경)

* **ML과 LLM의 분리:** LLM이 만능처럼 보이지만, 수치형/정형 데이터 기반의 의사결정 문제는 여전히 전통적인 ML(Scikit-learn, XGBoost 등)이 훨씬 강력하고 효율적입니다.
* **기술 혼동 방지:** 연구소 내에서 "모든 문제에 LLM을 붙이면 해결되겠지"라는 오해를 바로잡고, 문제의 본질에 따라 기술 스택을 결정하는 가이드를 제시하고자 합니다.

### 🎯 What (무엇을?)

Kaggle의 서로 다른 두 트랙을 병행 학습하여 **데이터 성격에 따른 기술 대응력** 확보

* **Track A (Traditional ML):** 정형 데이터 기반의 예측 모델링 (Classification/Regression)
* **Track B (LLM/NLP):** 텍스트 데이터 처리 및 LLM 프롬프트 엔지니어링/Fine-tuning

### 💡 Why (왜?)

실제 데이터 문제에서 **"이 문제는 ML로 풀 것인가, LLM으로 풀 것인가?"**를 판단하는 안목을 기르기 위함입니다. 무조건 최신 기술을 쓰는 것이 아니라, 비용 대비 효율이 가장 높은 솔루션을 도출하는 '엔지니어링 사고' 과정을 공유하고자 합니다.

### 🏁 Goal (목표)

* **ML 파트:** Tabular 데이터 기반 'Getting Started' 대회 상위 20% 수준의 성능 도달 및 Feature Engineering 노하우 정리.
* **LLM 파트:** NLP 관련 대회(예: 텍스트 분류, 요약, 채점 등) 참여를 통해 LLM의 강점과 한계(Hallucination, 비용 등) 파악.
